## 5. ACID & 崩溃恢复验证

### 5.1 单语句原子性（autocommit）

**关键测试** (`tests/integration/test_autocommit.py`)：

- `test_failed_insert_auto_rolls_back` — 多行 INSERT 中途失败，**整个语句**回滚，无 partial state
- `test_drop_table_rolls_back_on_failure` — DROP 失败恢复后表仍存在
- `test_pk_violation_does_not_leak_partial_data` — PK 冲突中所有行被丢弃

### 5.2 WAL 耐久性

- `tests/integration/test_acid.py` — COMMIT 后 `fsync`，power loss 不会丢失已提交事务
- `tests/integration/test_crash_recovery.py` — open 时检测 `*.wal` sidecar，启动 `Recovery.replay`
- `tests/integration/test_recovery_fuzz.py` — fuzz：随机 kill -9 模拟后验证 replay 一致

### 5.3 B+tree 在事务中

- `tests/integration/test_btree_in_transaction.py` — 事务内 split/merge，rollback 时索引页正确释放
- `tests/integration/test_ddl_in_transaction.py` — DDL 在事务中：CREATE/DROP 在 COMMIT 后生效，ROLLBACK 后消失

### 5.4 异常捕获保证

```python
try:
    db.execute("INSERT INTO t(id) VALUES (2**40)")  # INT overflow
except tinydb.errors.CodecError as e:
    assert "INT out of range" in str(e)
```

→ 所有列类型边界失败都进入同一条 `except CodecError` 分支。

## 6. codec 与 catalog 端到端

### 6.1 数据往返（end-to-end）

`tests/integration/test_parser_executor_roundtrip.py` 与 `tests/e2e/test_golden_sql.py` 共 45 golden-SQL 路径，覆盖：

- CREATE TABLE（含 PRIMARY KEY、UNIQUE、NOT NULL）
- INSERT（含类型化字面量：DATE / TIME / TIMESTAMP / DECIMAL）
- SELECT `*` / 列投影 / WHERE 复合条件 / ORDER BY ASC/DESC
- UPDATE / DELETE（WHERE 限制）
- BEGIN / COMMIT / ROLLBACK
- 聚合（COUNT/AVG/GROUP BY）—— 由 `tinydb-aggregation` 引入
- 索引路径 SELECT（`test_select_uses_index.py`）

每个 golden SQL 跑完对 stdout/stderr/snapshot 三件套断言。

### 6.2 schema 升级路径

- v1 文件（旧 `[name, type]` 数组形态）→ v2 `Column.to_dict()` 对象形态
- v2 schema (engine-v2) → v3 schema (engine-v1 + acid)
- 每条升级路径都有显式测试 + 升级失败测试（带 WAL sidecar → SchemaMismatch）

## 8. 性能基线（reference，非 gate）

来自 `tests/integration/test_constraints_perf.py`：

- 单 INSERT（autocommit, fsync）: ~3 ms（warm cache, NVMe）
- 多行 INSERT 100 行: ~30 ms
- 简单 SELECT 10k 行: < 50 ms（in-memory index 未命中时）
- 事务 COMMIT: ~5 ms
- WAL replay 1000 transactions: ~80 ms

> 上述数字仅作 reference，**不作为 CI gate**——MVP 阶段无 SLO。
